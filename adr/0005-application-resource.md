# ADR-0005: Application Resource

## Context

CloudOS has a powerful execution substrate — Resource Engine, Controller Runtime,
Workflow Engine, Workflow Service — but no first-class representation of the
user's **software**. The platform can manage projects, namespaces, workflows, and
executions, but it cannot answer the question: "What application does this user
want to run?"

Meanwhile, the architecture has grown from **infrastructure primitives** to a
level where we can build the first real product: deploying an application.

We need a new resource that represents the user's software in a way that is:

- **Provider-agnostic** — the same Application resource works for a React SPA,
  a Laravel API, a Python background worker, or a static site
- **Workflow-native** — every deployment is a workflow execution, not an ad-hoc
  script
- **Project-scoped** — applications belong to projects, just like every other
  resource
- **Observable** — status, health, conditions, and deployment history are
  first-class

## Decision

### What is an Application?

An Application is a **first-class CloudOS Resource** that represents the user's
software. It is NOT Docker. It is NOT Kubernetes. It is the user's code:

```
Application (React app, Laravel API, Next.js site, Python worker, etc.)

    ├── source     → where the code lives (git repo, local path)
    ├── runtime    → what runs it (node, python, go, static, docker)
    ├── build      → how to build it (command, output)
    ├── env        → environment variables
    └── deploy     → how to deploy it (port, domain, replicas)
```

An Application is to CloudOS what a `Deployment` is to Kubernetes — the
desired-state declaration of what should be running.

### Why is it a Resource?

Every managed object in CloudOS is a Resource. Resources get:

- **Uniform API** — CRUD via `GET /api/v1/resources/Application/{id}`
- **Validation** — schema enforcement at the Resource Engine level
- **Watching** — `?watch=1` for live updates
- **Event publishing** — lifecycle events on create/update/delete
- **Persistence** — survives kernel restarts

Making Application a Resource means it integrates with everything else the
platform already provides: the Dashboard can list applications, the SDK can
create them, controllers can reconcile them.

### Why does it use the Workflow Engine?

The key architectural insight: **the controller does not deploy directly**.

Instead, the Application Controller:

1. Validates the Application spec
2. Generates a **WorkflowDefinition** — a DAG of deployment steps
3. Submits the workflow via the **Workflow Service**
4. Updates Application status as the workflow progresses

This means every deployment becomes just another **WorkflowExecution Resource**.
The user gets:

- **Retry** — failed deployments can retry failed nodes
- **Replay** — re-run a completed deployment
- **History** — every deployment is a persistent WorkflowExecution
- **Observability** — progress, conditions, node-level status

### Why doesn't it deploy directly?

Separation of concerns:

| Layer | Responsibility |
|---|---|
| Application Resource | Desired state ("run my React app") |
| Application Controller | Validate + create deployment workflow |
| Workflow Engine | Execute the deployment steps |
| Providers | Actually run the code (local, Docker, cloud) |

If the controller deployed directly, we would lose:

- **Retry** — no built-in retry with backoff
- **History** — no persistent execution records
- **Observability** — no node-level progress tracking
- **Reliability** — no pause/resume/cancel
- **Uniformity** — every controller would need its own deployment logic

### How does it interact with Projects?

```
Project
    │
    ├── Application "frontend" (React SPA)
    │     └── WorkflowExecution "deploy-001"
    ├── Application "backend" (Laravel API)
    │     └── WorkflowExecution "deploy-001"
    └── Application "worker" (Python)
          └── WorkflowExecution "deploy-001"
```

Applications belong to Projects via `OwnerReference`. The Project controller
is responsible for tracking application count and health.

### How does it interact with Providers?

When a deployment workflow runs, its steps interact with **Providers** through
the existing provider abstraction:

```
Workflow Step: "Deploy Artifact"
    │
    Action: "provider.execute"
    │
    Provider: "local" → runs the app as a local process
    Provider: "docker" → runs the app in a container
    Provider: "aws" → deploys to AWS (future)
```

The Application Resource does not know about providers. The Workflow
Definition generated by the controller maps deployment steps to provider
actions. This keeps Application provider-agnostic.

### Application Lifecycle

```
Creating ──→ Running ──→ Stopped ──→ Deleting
  │            │            │
  └── Failed   └── Failed   └── Failed
```

- **Creating**: initial state; controller validates and starts first deployment
- **Running**: application is deployed and healthy
- **Stopped**: application is not running (paused, scaled to zero)
- **Failed**: deployment failed or health check is failing
- **Deleting**: cleanup in progress

### Deployment Strategy

When an Application is created or its spec changes:

1. Controller validates the Application spec
2. Controller creates a WorkflowDefinition with deployment steps:
   - Validate source
   - Build artifact (if applicable)
   - Deploy artifact via provider
   - Health check
   - Update URL / status
3. Controller submits the workflow to Workflow Service
4. WorkflowExecution Resource is created
5. Controller updates Application status: phase, url, health
6. Workflow progresses; each node updates WorkflowExecution status
7. On completion, controller marks Application as "Running"

### Conditions

The Application Resource uses Kubernetes-style conditions:

| Condition | When |
|---|---|
| `Created` | Application resource is initialized |
| `Deploying` | A deployment workflow is running |
| `Running` | Application is deployed and passing health checks |
| `Failed` | Deployment or health check failed |
| `Stopped` | Application is intentionally stopped |
| `Deleting` | Cleanup in progress |

## Alternatives Considered

### Direct deployment (no workflow)

The controller calls a provider directly when the Application is created.

Rejected because:
- Loses all workflow benefits (retry, replay, history, observability)
- Every controller reinvents deployment orchestration
- No uniform execution model

### Application as a workflow alias

Make Application just a label on a WorkflowDefinition — no separate resource.

Rejected because:
- Loses the Resource interface (uniform CRUD, validation, watching)
- Cannot express Application-specific fields (source, runtime, environment)
- Harder to build a dashboard around

### Docker-first

Define Application as a Docker container configuration.

Rejected because:
- CloudOS should be language/framework-agnostic
- Many users don't use Docker for development
- Provider-neutral means the same Application can deploy locally,
  via Docker, or to a cloud provider without spec changes

### Kubernetes-native

Make Application a Kubernetes CRD.

Rejected because:
- CloudOS is not Kubernetes; it should not depend on Kubernetes CRDs
- Kubernetes CRDs bring heavyweight dependencies
- CloudOS's Resource Engine already provides what we need

## Consequences

### Positive

- **First real product**: CloudOS can now deploy software, not just manage
  infrastructure
- **Provider-agnostic**: the same Application works with local, Docker, or
  cloud providers
- **Workflow-native**: every deployment gets retry, replay, history, and
  observability for free
- **Uniform UX**: Applications are Resources — they use the same API,
  validation, watching, and dashboard as everything else
- **Extensible**: new runtime types (Rust, Go, Java, Elixir) are just
  additions to the runtime enum
- **Intent-ready**: "Deploy my React app" maps naturally: Intent → Workflow →
  Application Resource

### Negative

- **More moving parts**: Application → Controller → Workflow Service →
  Workflow Engine → WorkflowExecution → Provider is a deep chain
- **Workflow coupling**: Applications depend on the Workflow Engine being
  healthy; if the engine is down, applications cannot deploy
- **First deployment latency**: the workflow adds overhead vs a direct
  deploy call, though this is negligible for production use

### Migration

1. Create the Application resource type, controller, and tests
2. Register the Application kind and controller in kernel boot
3. Build a "local" provider that runs applications as processes
4. Add source code integration (git clone, build)
5. Wire up the Dashboard to show applications and their deployments
